# 穿搭闖關介面規格書（修正版）

## 1. 文件目的

本文件定義「穿搭小達人」任務關卡的畫面文案、題庫篩選規則與操作元件調整，供 UI、前端與關卡內容製作使用。

## 2. 畫面架構

| 區域 | 功能 |
|---|---|
| 頂端工具列 | 首頁、說明、音效開關與計時器。 |
| 左側任務卡 | 顯示關卡進度、任務情境與場景圖。 |
| 中央角色區 | 呈現角色與已選穿搭的即時預覽。 |
| 左側部位清單 | 顯示身體部位相關資訊；任務卡可刻意覆蓋部分按鈕。 |
| 右側衣櫃 | 依上衣、下身、鞋子、配件分類，供玩家選擇服裝。 |
| 右下操作區 | 顯示搭配完成進度，並提供重置與跳題操作。 |

## 3. 本階段玩法與系統保留範圍

新版採用「截圖版介面 + 舊版遊戲核心邏輯」的整合方式。畫面改為情境任務式換裝，但保留既有遊戲中已驗證的題庫與關卡流程。

### 3.1 必須保留的舊版邏輯

- **題庫篩選**：沿用既有題庫的隨機出題與情境標籤篩選機制。
- **服裝驗證**：沿用依題目情境、服裝類別與規則判斷搭配是否正確的驗證邏輯。
- **重置服裝**：清除當前題目已選、已套用的服裝，回到本題預設角色狀態。
- **跳過這題**：可跳過當前關卡；跳題後以既有規則記錄關卡結果並進入下一題。
- **結果回饋**：完成驗證後，保留正確／不正確的結果提示、錯誤原因與重試或下一題的流程。

### 3.2 截圖版衣櫃呈現方式

- 右側衣櫃採截圖版的卡片式圖像介面。
- 分類固定為：`上衣`、`下身`、`鞋子`、`配件`。
- 各分類以圖示分頁切換；衣物以等尺寸格狀卡片顯示。
- 點選卡片後，角色須立即預覽該服裝，且卡片需顯示明確選取狀態。
- 不使用舊版的「全部」分類分頁作為主要入口。

### 3.3 預先染色的衣物卡與顏色驗證

本階段不提供玩家手動選擇色票或進行染色；顏色改為預先製作在衣物卡上的視覺與資料屬性。

- 移除主畫面的染色面板與色票選擇操作。
- 每張衣物卡應包含「服裝款式」與「預先染好的顏色」；角色預覽必須顯示該顏色。
- 服裝驗證以「服裝 + 顏色」的組合進行。例如題目答案為「烏色个長褲」時，只有預先染為烏色的長褲屬於正確答案；同款但不同顏色的長褲可作為誘答。
- 原有色彩資料、標籤與驗證規則應保留並轉用於生成衣物卡，勿刪除。

#### 3.3.1 每關保證可解的目標套組

每一關生成衣櫃時，系統必須先放入完整的「目標套組」，再補入誘答卡。目標套組是該關取得 10 分所需的所有服裝與配件。

- 依關卡設定的必要部位，為每個必要部位各放入 1 件正確的預先染色目標衣物。
- 基本必要部位為：上衣、下身、鞋子；題庫若另指定頭部、頸部、腿部或其他配件，也必須一併放入正確目標卡。
- 所有目標卡必須在同一關、同一時間可見或可透過分類分頁取得；不可因分頁、隨機抽樣或數量上限而被排除。
- 玩家同時選到全部必要部位的正確目標卡後，提交驗證必須能取得 10 分。
- 目標套組中的任一張卡均不可被隨機誘答流程替換、覆蓋或刪除。

#### 3.3.2 誘答卡生成

- 在完整目標套組已保留後，再從衣物資料庫隨機抽取誘答卡。
- 誘答可隨機改變服裝款式、部位與顏色；包含「款式錯誤、顏色錯誤、情境不符」等類型。
- 可加入同款不同色的衣物，作為顏色辨識誘答；但不得造成目標卡無法辨識或無法選取。
- 誘答數量須受每個分類頁面的可視容量限制；若空間不足，優先保留全部目標卡，再減少誘答數量。
- 同一局可使用可重現的亂數種子，以利除錯與測試；一般遊玩時每關的誘答組合可不同。

### 3.4 關卡出題邏輯

每局共 10 題，分為兩個學習階段。兩階段皆從題庫中隨機抽取符合條件的題目；同一局內不得重複抽到相同題目。

| 關卡 | 題型 | 出題規則 |
|---|---|---|
| 第 1–5 題 | 情境穿搭題 | 從題庫隨機抽題，以中文顯示完整的情境、顏色與衣物資訊。玩家依題目完成穿搭。 |
| 第 6–10 題 | 拼音辨識穿搭題 | 維持與第 1–5 題相同的隨機抽題、情境與服裝驗證邏輯；每題再隨機選擇「顏色」或「衣物」其中一項，以客語拼音取代中文顯示。另一項仍以中文呈現。 |

後半段題型範例：

- 顏色拼音題：`vuˊ sedˋ 个 長褲，看螢火蟲也不怕被蚊子叮。`
- 衣物拼音題：`烏色个 chhongˇ fuˋ，看螢火蟲也不怕被蚊子叮。`

實作規則：

- 第 6–10 題的「拼音欄位」每題以 50% 機率決定為顏色或衣物；不得同時將兩者改為拼音。
- 題目在畫面上只顯示選定的語言形式，不額外同時附上中文答案，避免直接洩題。
- 驗證仍依題庫中的標準服裝／顏色資料進行，不因前端顯示為拼音而改變答案判斷。
- 第 6–10 題可將顏色或衣物其中一項改為拼音；顏色雖不由玩家手動染色，仍需透過選取正確的預先染色衣物卡完成驗證。

## 4. 已確認修正需求

### 3.1 左側部位按鈕遮擋

- 任務卡覆蓋左側部位按鈕為既定視覺設計，不需調整版面或避開遮擋。
- 被遮住的按鈕僅需保留「客」字可見，作為客庄主題的視覺提示；其餘文字遭遮擋可接受。

### 3.2 任務文案與題庫篩選

任務文字以單一 `div` 呈現，文案如下：

> 著藍衫，體驗做檔案和藍染，像是穿越到早時客庄。

文案必須維持在同一個 `div`，不可拆成不同欄位或獨立元件。

題庫篩選規則：

| 文案片段 | 篩選用途 |
|---|---|
| `著藍衫` | 用於隨機篩選題庫。 |
| `，體驗做檔案和藍染，像是穿越到早時客庄。` | 同樣作為題庫篩選條件。 |

實作注意事項：

- 篩選邏輯可解析同一段文字中的前、後半句，但畫面上仍以完整句子一次顯示。
- 篩題結果應能對應此關卡所需的服裝類型與任務情境。

### 3.3 搭配進度文案

- 原文案：`穿搭 (0/1)`
- 新文案：`完成搭配 0/1`
- 數字依玩家已完成的搭配數量即時更新，例如：`完成搭配 1/1`。

### 3.4 重置按鈕文案

- 原按鈕：`重置`
- 新按鈕：`重置服裝`
- 操作行為維持不變：清除目前已選擇、已套用的服裝，讓角色回到此題預設狀態。

## 5. 介面優化建議

### 4.1 加強任務與選衣的連結

任務卡除情境敘述外，建議增列簡短、具體的行動提示，例如「請選擇適合藍染體驗的服裝」。這能讓玩家快速理解選擇目標，不必自行從故事文字推論。

### 4.2 提供明確的服裝選取回饋

衣櫃中被選取的服裝卡片應至少具備一種明確狀態：外框高亮、勾選圖示或底色變化。角色換裝預覽也應同步更新，使玩家知道選擇已生效。

### 4.3 補足服裝名稱或說明

建議在點選服裝後，顯示服裝名稱與簡短描述；可使用衣櫃下方提示區或懸浮提示。此項對兒童辨識服裝、理解情境與學習文化內容均有幫助。

### 4.4 降低誤按「跳過這題」的機率

「跳過這題」應比主要操作低調，例如採次要按鈕樣式，並在點擊後顯示確認提示。避免使用者在本意是重置服裝時直接跳過題目。

### 4.5 讓計時器狀態更清楚

計時器建議加上「計時中」或「剩餘時間」的標示，並在尚未開始時顯示「尚未開始」或 `00:00` 的靜止狀態，降低玩家誤解。

## 6. 驗收條件

- 任務卡遮住左側部位清單時，仍可看到「客」字，且不調整既有覆蓋設計。
- 任務完整文案在單一 `div` 內呈現。
- 系統可分別以「著藍衫」及逗號後文案作為題庫篩選依據。
- 進度按鈕文字顯示為「完成搭配 0/1」，並會隨完成數量更新。
- 重置按鈕顯示為「重置服裝」，且能正確清除本題已選服裝。
- 選取服裝後，衣櫃與角色預覽至少各提供一種可辨識的即時回饋。
- 題庫篩選、服裝驗證、重置、跳題與結果回饋皆可沿用舊版的關卡流程正常運作。
- 衣櫃以「上衣／下身／鞋子／配件」四分類、卡片式圖像格呈現。
- 主畫面不顯示色票或染色操作；顏色由衣物卡預先帶入，並納入服裝組合驗證。
- 每關的衣櫃都必須完整提供該關所有必要部位的正確目標衣物卡；選齊目標套組後可取得 10 分。
- 每局前 5 題為隨機中文情境穿搭題；後 5 題為相同邏輯的拼音辨識穿搭題。
- 第 6–10 題每題僅將顏色或衣物其中一項改以客語拼音顯示，且不重複抽取同一題。

## 7. 前導故事

### 7.1 範圍與操作

- 遊戲開始前顯示前導故事；畫面固定為 1920×1080，禁止產生垂直捲軸。
- 玩家可按「繼續」逐段閱讀，或按「跳過故事情境」直接前往遊戲說明／腔調選擇與開始遊戲畫面。
- 跳過只略過劇情，不影響題目、計時、分數或排行榜資格。
- 前導故事採「場景背景 + 角色／物件圖 + 半透明對話框」的可配置結構；程式不可將對話或圖片路徑硬寫在元件內。
- 故事對話、說話者名稱與排行榜五星稱號／結算評語本階段固定使用中文，**不納入多語或多腔調資料規格**。客語腔調切換僅影響題目中的客語單字、拼音與未來另行指定的題句內容。

### 7.2 故事場景資料與假圖策略

正式美術尚未交付時，必須先以可替換的假圖讓前導流程完整可玩。假圖僅用於版面、點擊、轉場與劇情驗收，不可作為正式上線美術。

| scene_id | 故事內容 | 暫用資產槽位 | 正式美術交付需求 |
|---|---|---|---|
| `intro_01_ready` | 阿梅開心表示已準備出發。 | `placeholder_intro_01_bg`、`placeholder_ame_happy` | 前導背景、阿梅開心立繪。 |
| `intro_02_mom_warning` | 媽媽提醒須依天氣與場合穿得舒服、體面。 | `placeholder_intro_02_bg`、`placeholder_mom_surprised`、`placeholder_ame_normal` | 場景背景、媽媽驚訝立繪、阿梅一般立繪。 |
| `intro_03_wrong_examples` | 阿梅想像穿錯衣服／顏色的尷尬情境，並請玩家幫忙。 | `placeholder_intro_03_bg`、`placeholder_ame_worried`、`placeholder_bubble_hot_puffer`、`placeholder_bubble_wedding_black` | 阿梅困擾立繪、炎熱穿羽絨錯誤示範泡泡、婚禮全黑錯誤示範泡泡。 |
| `intro_04_to_closet` | 轉場至衣櫃與正式關卡。 | `placeholder_closet_bg` | 衣櫃／遊戲場景背景。 |
| `result_01_leaderboard` | 通關後顯示結算與排行榜。 | `placeholder_result_bg` | 成績結算／排行榜背景。 |

資產實作規則：

- 以前綴式資產 ID 與設定檔對應圖檔；正式圖完成後只替換檔案或設定值，不修改劇情流程程式。
- 假圖需清楚標示場景與角色名稱，維持正式圖的預定比例與站位，避免美術替換時重做排版。
- 正式美術以 `參考風格` 資料夾中的手繪蠟筆質感、暖色塊、粗深色描邊與童趣紙感為方向；實際生成／繪製前再依該資料夾確認角色比例與色彩。
- 每張正式圖交付時需註明：資產 ID、透明背景與否、建議顯示尺寸、角色站位（左／中／右）及是否可和其他圖層重疊。

#### 7.2.1 背景圖尺寸與安全範圍

- 前導故事、衣櫃轉場、結算與排行榜等全畫面背景一律採 **16:9**。
- 程式設計基準為 `1920 × 1080`；美術正式交付建議使用較高解析的 `2400 × 1350` PNG，必要時另輸出 WebP 供網頁載入。
- `2400 × 1138` 約為 `2.11:1`，不是 16:9，不得作為全畫面背景的正式交付尺寸；否則在 16:9 裝置會產生裁切或上下留白。
- 所有關鍵角色、文字、道具與互動提示需保留至少畫面四周 10% 的安全邊界；背景邊緣僅放可裁切的裝飾元素。
- 角色立繪、錯誤示範泡泡與前景道具獨立交付透明背景 PNG，透過程式疊加在 16:9 背景上，不烘焙進背景圖。

### 7.3 前導故事驗收條件

- 不使用正式美術時，五個場景仍可依序播放、可跳過，並可順利進入遊戲。
- 替換任一資產 ID 對應圖檔後，不需要修改劇情或互動程式。
- 對話框在所有場景皆清楚可讀，且不遮住「繼續」與「跳過故事情境」。

## 8. 排行榜與 API 交接規格

### 8.1 本階段責任切分

目前前端無權限讀取正式排行榜資料，因此先完成「排行榜畫面 + Mock 資料 + API 介面層」。在未串接正式 API 前，遊戲必須仍可完整試玩、結算與顯示假排行榜。

- 畫面元件不得直接呼叫特定 API URL；統一透過 `LeaderboardService` 取得資料與送出成績。
- 以設定值切換 `mock` 與 `api` 模式；開發與美術驗收預設使用 `mock`。
- 正式上線時，分數驗證、名次計算與寫入排行榜必須由後端負責；前端不得以自行計算的名次作為權威結果。
- 無 API 權限、載入中、排行榜空白、讀取失敗時均需有對應畫面與可理解文案，且不可阻斷玩家查看本局結算。

### 8.2 排行榜排序與顯示規則

- 排名排序：`score` 由高至低 → `elapsed_ms` 由低至高 → `submitted_at` 由早至晚。
- 計時採正計時，顯示格式為 `HH:MM:SS.mmm`；排行榜儲存毫秒整數 `elapsed_ms`。
- 顯示欄位：排名、遮蔽後的玩家名稱、分數、完成時間；另顯示總參加人數與總遊玩次數（若 API 有提供）。
- 玩家名稱由後端或平台傳回已遮蔽的 `display_name`；前端不得自行取得或揭露完整姓名。
- 結算頁顯示本局分數、耗時、名次與五星稱號／中文評語。五星百分比區間依內容規格執行；正式百分比與名次以後端回傳為準。
- 參考平台的「前幾名得獎」規則不直接套用；本遊戲以既定的五星稱號與百分比分級為主。若未來需增加獎項名次，另行定義。

### 8.3 前後端 API 契約草案

**取得排行榜（建議）**

`GET /api/events/{eventId}/leaderboard?limit={limit}`

```json
{
  "eventId": 9,
  "rankingRule": "score_desc_elapsed_ms_asc_submitted_at_asc",
  "participantCount": 53,
  "playCount": 120,
  "entries": [
    {
      "rank": 1,
      "displayName": "王○○",
      "score": 100,
      "elapsedMs": 34331,
      "submittedAt": "2026-07-20T10:00:00Z"
    }
  ],
  "myEntry": {
    "rank": 12,
    "score": 80,
    "elapsedMs": 52110,
    "starLevel": 3,
    "titleId": "fashion_observer"
  }
}
```

**送出單局結果（建議）**

`POST /api/events/{eventId}/game-results`

前端只送出完成一局所需的識別與作答紀錄；後端應依題目、驗證結果與伺服器時間重新驗算分數與耗時，再回傳最終名次。不可只信任前端傳入的總分或名次。

```json
{
  "runId": "uuid",
  "questionSetId": "uuid-or-seed",
  "answers": [
    { "questionId": "q01", "selectedOutfitIds": ["clothes:short_shirt@yellow"] }
  ]
}
```

### 8.4 Mock 資料與工程師交接清單

- 前端先建立與上述回應完全相同結構的 `leaderboard.mock.json`，讓排行榜與結算頁可在沒有 API 的環境中測試。
- `LeaderboardService` 至少提供：`getLeaderboard(eventId, limit)`、`submitGameResult(eventId, payload)`；mock 與正式 API 的回傳型別必須一致。
- API 工程師交接前須確認：認證方式、`eventId` 來源、是否只保留個人最佳單局、排行榜統計範圍、百分比／星級由何端計算，以及 API 錯誤碼。
- 前端驗收需涵蓋：正常榜單、無資料、載入失敗、玩家未上榜、同分不同時間、同分同時間五種情境。

### 8.5 排行榜驗收條件

- 未串接 API 時，使用 mock 仍可完成一局並看到結算與排行榜。
- 切換至正式 API 後，排行榜元件不需要更改畫面程式或資料欄位名稱。
- 顯示順序、遮蔽名稱與時間格式符合本節規則；前端不自行覆寫後端回傳名次。
